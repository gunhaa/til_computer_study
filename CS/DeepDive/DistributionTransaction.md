# Distribution Transaction

> https://www.youtube.com/watch?v=DOFflggE_0Q

- 서비스 시작시에는 일반 Lock, DB를 사용해서 해결 가능하다
  - 하지만 서비스가 확장되고 분산 모델로 바뀌게 되어, 각 서비스마다 사용하는 DB가 달라지게 되면 여기서부터 분산락이 필요해진다
- 분산락의 역사는 2PC - (SAGA) Choreography/Orchestration로 발전했다
   - 현재의 큰 서비스들은 사실상 Orchestration패턴 혹은 분산 트랜잭션이 지원되는 DB를 사용한다

## Two-Phase Commit (2PC)

- 분산 데이터베이스 환경에서 모든 노드가 트랜잭션을 확정(Commit)하거나 취소(Rollback)하도록 보장하는 전통적인 합의 프로토콜
- 이 패턴은 Coordinator가 락이 필요한 곳에 락을 요청 한 후, 레코드 락을 건 뒤 트랜잭션을 동시에 진행하는 방식이다
  - 1.Prepare 단계: 제어자가 모든 참여자에게 "커밋할 준비가 되었는가?"라고 묻는다. 참여자들은 리소스에 락(Lock)을 걸고 준비 완료 응답을 보낸다
  - 2.Commit 단계: 모든 참여자가 "Yes"라고 답하면 제어자가 커밋 명령을 내리고, 하나라도 "No"라고 답하면 모든 참여자에게 롤백 명령을 내린다
- 분산 락 초기에 사용되었지만, 빠르게 사장되었다
  - 블로킹 문제: 모든 참여자가 응답할 때까지 리소스 락이 유지되므로, 가장 느린 노드의 속도에 전체 시스템 성능이 맞춰진다
  - 단일 장애점: 제어자가 1단계와 2단계 사이에서 장애가 발생하면, 모든 참여자는 결정을 내리지 못한 채 락을 들고 무한정 대기하게 된다
  - 확장성 저하: 서비스와 노드 수가 늘어날수록 통신 비용과 대기 시간이 기하급수적으로 증가하여 현대적 마이크로서비스 아키텍처에는 적합하지 않다

## Saga Pattern (사가 패턴)

- 2PC의 단점(Blocking)을 해결하기 위해 업계(Uber, Netflix 등)에서 표준적으로 사용하는 방식이다
- 핵심 원리
  - 하나의 큰 트랜잭션을 여러 개의 독립적인 로컬 트랜잭션 체인으로 나눈다
- 보상 트랜잭션 (Compensating Action)
  - 이전 단계로 되돌리는 'Rollback' 대신, 이미 커밋된 결과를 비즈니스적으로 취소하는 작업을 수행한다
- 일관성 모델
  - 최종 일관성(Eventual Consistency)을 지향한다
  - 중간 과정에서 데이터가 일치하지 않는 순간이 있을 수 있지만, 결국에는 올바른 상태로 수령한다

### Choreography

- 방식: 중앙 제어자 없이 각 서비스가 이벤트를 발행(Publish)하고 구독(Subscribe)하며 연쇄적으로 반응하는 방식이다
  - 장점: 단순한 구조에서 구축하기 쉽고 서비스 간 결합도가 낮다
  - 단점: 단계가 많아지면(5~6개 이상) 전체 흐름을 파악하기 어렵고, 어디서 문제가 생겼는지 추적(디버깅)하기 매우 힘들다
- 이 방식으로 진행이 어려울 정도로 로직이 복잡해지면, Orchestration방식으로 리팩토링을 진행하는게 일반적인 흐름이다

### Orchestration (오케스트레이션)

- 방식: 중앙 오케스트레이터(Orchestrator) 서비스가 전체 흐름을 제어하며, 각 서비스에 할 일을 지시하고 응답을 기다린다
  - 장점: 복잡한 비즈니스 로직과 분기 처리에 적합하며, 현재 상태를 한눈에 파악할 수 있으며, 오케스트레이터가 가용성을 유지하면 장애 복구도 원활하다
- 엔진: Temporal, AWS Step Functions 등을 사용한다
- 해당 방식은 구현이 복잡해서, 검증된 엔진을 사용하는 것이 더 낫다(Temporal)

### Transactional Outbox Pattern

- 분산 환경에서 (`DB 저장`과 `이벤트 발행`)이라는 두 작업의 원자성을 보장하기 위한 패턴이다
- 문제점: DB 저장은 성공했는데 네트워크 문제로 이벤트 발행만 실패할 경우, 후속 서비스가 동작하지 않아 데이터가 꼬이게 된다
- 해결책: 로컬 DB 내에 Outbox라는 전용 테이블을 만들고, 비즈니스 데이터와 발행할 이벤트를 하나의 로컬 트랜잭션으로 묶어 저장하며, 이후 별도의 프로세스가 이 테이블을 읽어 메시지 브로커로 전달한다

## Trade-off

- 분산 트랜잭션은 가급적 안 만드는 것이 최선의 방법이다
  - 가능하다면 서비스 경계를 잘 설계하여 하나의 DB 안에서 로컬 트랜잭션으로 해결하는 것이 가장 빠르고 안정적이다
- 강한 일관성이 꼭 필요한 경우는 직접 2PC를 구현하기보다는 Google Spanner나 Cockroach DB 같은 분산 트랜잭션을 자체 지원하는 DB를 사용하는 것이 좋다
- Saga(특히 Orchestration 방식)과 Transactional Outbox 패턴을 조합하여 구축하는 것이 현대적인 정석이다
