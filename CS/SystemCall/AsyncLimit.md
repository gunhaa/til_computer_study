# 비동기 시스템 콜(IOCP/ EPOLL/ KQUEUE)의 한계

- 기본적으로 비동기 시스템콜은 mode change를 한번만 실행시켜 context switching을 최소화하여 효율적으로 실행한다
  - kernel mode에서 확인할 수 있는 버퍼에 결과를 담으며, 완료 통지만 보내서 결과적으로 user -> kernel mode change 한번이다
    - 결국 recv(), read() 두번의 System call은 반드시 필요하다
- 위 원리 때문에 문제가 생길 수 있다
  - SSE의 경우 통지가 한번이 아닌 완료 전까지 하나의 요청이 많은 요청을 보낸다
  - SSE방식은 kernel config를 넣지 않은 비동기 시스템에서 문제가 생길 수 있다
  - 현재 LLM의 Response의 경우 대부분 이 방식이다
    - 하나의 요청에도 불구하고 수 백번의 read(), recv() 시스템 콜이 반복된다
    - 비동기의 장점인 'Mode Change 최소화'가 무색하게 수천 번의 모드 전환이 강제된다
    - 네트워크 카드로부터 들어오는 잦은 패킷 유입은 하드웨어 인터럽트를 유발하고, CPU는 실제 로직보다 인터럽트 처리에 더 많은 사이클을 낭비하게 된다
    - Async를 사용하는 싱글 스레드 기반 이벤트 루프들은 폭발적인 OS 레벨의 부하를 코어 하나에서 온전히 감당해야 하며, 이때 발생하는 CPU 점유율 상승은 로직의 복잡도가 아닌 '네트워크 스택 처리 비용'이다
