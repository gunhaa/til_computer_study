# Chapter 5. 작은 것으로 큰 성과 이루기, 캐시

- 문제: CPU의 속도와 메모리의 속도가 일치하지 않는다
  - 문제가 된 속도를 해결하기 위해 캐시를 사용한다(register, L1, L2, L3)
    - 멀티코어 환경에서 캐시를 사용하면 코어 간 일관성 문제가 발생한다(CPU는 일관성을 유지하기 위해 MESI protocol을 사용하여 성능을 희생한다)
    - 소프트웨어적으로 데이터 가시성과 일관성을 완벽히 맞추기 위해서는 CPU 성능 최적화를 제한해야 하므로 속도를 희생해야 한다(명령어 재정렬 및 캐시 지연 쓰기를 방지하기 위해 고급 언어에서 volatile 키워드나 Memory Barrier 사용)
- (반드시)캐시 시간/공간적 지역성을 최적화 하기 전 반드시 분석도구를 사용하여 캐시의 적중률이 시스템의 병목이 될때만 최적화를 해야한다

## Section 5.3. 다중 스레드 성능 방해자

- 캐시는 `캐시 라인`이라는 이름으로 저장된다
  - `캐시 라인`이 캐시의 시간/공간적 지역성이 생기는 이유이다
  - `캐시 라인`의 묶음은 일반적으로 64byte이며, 캐시가 적중하지 못하면 이 묶음 데이터가 캐시에 저장된다

### 5.3.2 두 번째 성능 방해자: 캐시 튕김 문제

- 첫 번째 예제 프로그램
```cpp
atomic<int> a;

void threadf()
{
    for (int i = 0; i< 500000000; i++)
    {
        ++a;
    }
}

void run()
{
    thread t1 = thread(threadf);
    thread t2 = thread(threadf);

    t1.join();
    t2.join();
}
```

- 두 번째 예제 프로그램
```cpp
atomic<int> a;

void run()
{
    for (int i = 0; i < 1000000000; i++)
    {
        ++a;
    }
}
```

- 첫 번째 프로그램은 16초, 두 번째 프로그램은 8초가 걸린다
- linux의 perf를 이용해 분석해보면, 첫 번째 프로그램은 `0.15 insn per cycle`, 두 번째 프로그램은 `0.60 insn per cycle`이다
  - 첫 번째 프로그램은 하나의 클럭 주기 동안 기계 명령어가 0.15개 실행됬고, 두 번째 프로그램은 하나의 클럭 주기 동안 0.60개의 기계 명령어가 실행되었다
  - 다중 코어 컴퓨터를 사용했기에, 서로의 코어에 a를 사용하고 이것이 캐시되고 있어 atomic한 연산을 위해 a변수를 반드시 invalidation해야한다
    - 이로 인해 캐시 튕김이 발생하며, 다중 스레드 프로그램에서 더 많은 시간이 걸리게 된다
    - 이를 캐시 튕김(cache line bouncing) 혹은 캐시 핑퐁(cache ping-pong)이라고 한다

### 5.3.3. 두 번째 성능 방해자: 거짓 공유 문제

- 첫 번째 예제 프로그램과 두 번째 예제 프로그램을 위한 전역 변수 선언

```cpp
struct data
{
    int a;
    int b;
}
struct data global_data;
```

- 첫 번째 프로그램

```cpp
void add_a()
{
    for (int i = 0; i < 500000000; i++)
    {
        ++global_data.a;
    }
}

void add_b() 
{
    for (int i = 0; i < 500000000; i++)
    {
        ++global_data.b;
    }
}

void run()
{
    thread t1 = thread(add_a);
    thread t2 = thread(add_b);

    t1.join();
    t2.join();
}
```

- 두 번쨰 프로그램

```cpp
void run() 
{
    for (int i = 0; i < 500000000; i++)
    {
        ++global_data.a;
    }
    
    for (int i = 0; i< 500000000; i++)
    {
        ++global_data.b;
    }
}
```

- 첫 번쨰는 다중 스레드 프로그램, 두 번째는 싱글 스레드 프로그램이다
- 하나의 변수를 공유하지 않음으로 캐시 튕김 문제가 없을 것이라고 생각할 수 있다
- 하지만 첫 번째 다중 스레드 프로그램은 3초, 두 번째 싱글 스레드 프로그램은 2초가 걸린다
  - 이 경우 같은 변수를 공유하지 않더라도, 캐시의 핵심은 `캐시라인`이다
  - 변수가 다르더라도 동일한 캐시라인을 공유하고 있을 수 있다
  - a변수에 접근할 때 캐시가 접근하지 않으면 a변수가 포함된 캐시라인이 캐시에 저장되는데, 이 때 b변수도 캐시에 함께 저장될 확률이 매우 높다
  - 이 상황을 거짓 공유(false sharing)이라고 한다
  - 해결 방법은 의외로 간단하다. 두 변수 사이에 사용되지 않는 데이터를 채우면 된다
  ```cpp
  struct data
  {
    int a;
    // 4byte * 16 = 64byte padding
    int arr[16];
    int b;
  }
  ```

#### atomic변수가 아니라 거짓 공유여도 문제없는 것 아닌가?

- 어차피 다른 변수(a와 b)를 건드리고, atomic도 아니니까 서로 신경 안 쓰고 각자 캐시에 적어두었다가 나중에 메인 메모리에 보내면 되는 거 아닐까?
  - 같은 캐시라인이면 내가 사용하지 않는 것이라도 현재 상태를 검사한다
  - 캐시 일관성 프로토콜(MESI)를 통해서 검사한다
    - CPU는 데이터가 atomic인지 아닌지 관심이 없으며, 오직 "캐시라인 단위"로만 움직인다
    - 코어 1 (스레드 1): global_data.a를 올리기 위해 해당 캐시라인을 읽어와 값을 변경하며, 이때 이 캐시라인 상태는 Modified(수정됨)가 된다
    - 코어 2 (스레드 2): 동시에 global_data.b를 올리려고 하지만 코어 2가 보기에 코어 1이 가지고 있는 캐시라인이 'Modified' 상태이다
    - 캐시 튕김(Cache Bouncing): 캐시 일관성을 유지해야 하므로, 코어 2는 코어 1에게 "그 캐시라인 메인 메모리(또는 L3)에 쓰고 나한테 넘겨!"라고 요청하며, 코어 1의 캐시라인은 무효화(Invalid)되고, 코어 2가 캐시라인을 받아와서 b를 수정한다
    - 다음 루프에서 코어 1이 다시 a를 바꾸려고 하면, 방금 전 코어 2가 캐시라인을 가져갔으므로 또 뺏어와야 한다.
- 변수 a와 b는 독립적이라 race condition은 없지만, 하드웨어 구조상 같은 캐시라인에 묶여 있다는 이유만으로 두 코어가 하나의 캐시라인을 서로 뺏고 뺏기게 된다
- 같은 변수를 공유하면 race condition이 발생한다
  - MESI 프로토콜(하드웨어에서의 일관성 강제 유지 프로토콜)을 통해 코어간의 캐시는 언제나 최신화 시키지만, operation 자체가 atomic하지 않아서(read, +1, write 3개의 동작) race condition이 발생하는 것이다
  - 공유 변수를 사용해도 캐시라인의 일관성(Consistency)은 하드웨어 수준에서 실시간으로 강제되기 때문에 거짓 공유로 인한 성능 저하가 그대로 발생한다
- 이 과정에서 CPU의 동작이 증가하고 대기 시간이 길어져 멀티 코어가 싱글 코어보다 느려지게 된다

## Section 5.4. 봉화희제후와 메모리 장벽

> 명령어의 비순차적 실행은 멀티스레드 프로그래밍에서 많은 문제를 발생시키기에, 그 원리를 확실히 이해하고 사용해야한다

```cpp
bool is_enemy_coming = false;
int enemy_num = 0;

// 주유왕 스레드
void thread_zhouyouwang()
{
    enemy_num = 100000;
    is_enemy_coming = true;
}

// 제후 스레드
void thread_zhuhou()
{
    int n;

    if(is_enemy_coming)
    {
        n = enemy_num;
    }
}
```

- 이 스레드 두 개가 동시에 실행되면 n은 얼마인가?
  - 정답은 n = 100000일 것 같지만, 그렇지 않다
  - n은 값이 0이 될 수있다
  - 하지만 x86플랫폼에서는 n=0이 되는 일이 발생하지 않는다
- 위 예시의 더 직관적인 코드는 아래의 예시와 같다

```plaintext
// X와 Y의 초기 값은 0이다
thread1   thread2
 X = 1     Y = 1
 a = Y     b = X
```
- 위 코드와 같은 상황에서는 3가지 상황 밖에 없다고 생각하기 쉽다
  1. thread1이 먼저 실행되며, 이때 a=0, b=1이다
  2. thread2가 먼저 실행되며, 이때 a=1, b=0이다
  3. thread1, thread2가 동시에 실행되며 a=1, b=1이다
  4. 더 이상 다른 상황이 없다고 생각하기 쉽지만, x86에서는 a와 b모두가 0일수 있다
- 4번이 발생하는 이유는 `CPU는 반드시 엄격하게 프로그래머가 코드를 작성한 순서대로 기계 명령어를 실행하지 않기 때문`이다
  1. 기계 명령어를 생성하는 단계에서 컴파일 중 명령어를 재정렬한다
  2. CPU가 명령어를 실행 중 비 순차적으로 실행시킨다
- 이 것이 바로 비순차적 명령어 처리(Out of Order Execution, OoOE)이다
