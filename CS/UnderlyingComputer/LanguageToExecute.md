# Chapter1. 프로그래밍 언어부터 프로그램 실행까지, 이렇게 진행된다

## 프로그래밍 언어 bottom up

- CPU가 알아 들을 수 있는 것은 0, 1 뿐이다
  - e.g.1
    ```plaintext
    01010101110101
    01110101010101
    01000101111101
    10101011101011
    ```
  - 처음에는 binary만 사용해 프로그램을 만들다가, 개발의 생산성을 높이기위해 CPU에 친숙한 binary보다는 인간이 알아듣기 쉬운 어셈블리어를 개발한다
- 어셈블리어는 기계어와 해당 특정 작업을 간단하게 대응시켜 기계어를 인간이 일곡 이해할 수 있는 단어와 대응시켰다
  - e.g.2
  ```assembly
  sub $8, %rsp
  mov $.LC0, %edi
  call puts
  mov $0, %eax
  ```
  - 프로그래머가 이렇게 작성하면, 이에 대응하는 바이너리로 바꿔 CPU에서 실행시킨다(e.g.1)
  - 프로그래머는 1101.. 을 기억할 필요 없이 add, sub, mov처럼 인간이 인식할 수 있는 단어만 기억하면 된다
  - 어셈블리어는 인간이 인식할 수 있지만, 어셈블리어는 기계어와 마찬가지로 여전히 low-level language이다
  - 사용함에 따라 인간의 추상적인 표현을 CPU가 이해할 수 있는 구체적인 구현으로 변환할 수 있는 방법이 필요해지며, 이런 필요성에 따라 high-level language가 만들어졌다
- high-level language는 재귀 구조로 구현되어 있다
  - if/for/statement의 재귀 구조로 구현되어 있다
  - 이 언어에도 문제가 존재한다, CPU는 제조사에 따라 같은 명령이라도 다른 binary가 필요하다
    - C같은 컴파일 언어의 경우 이 한계를 극복하기위해 타겟 cpu마다 다른 컴파일러를 사용했다
    - 하지만 cpu마다 system call이 다른 경우도 많아서 타겟 cpu에 맞는 추가 포팅이 필요하다
      - 이 한계를 극복하기 위해 java같은 vm을 사용하는 언어가 나왔다
      - vm은 interpreter를 내장해, interpreter만 바꿔서 사용하면 원하는 cpu로 바로 binary를 바꿔 사용할 수 있다
      - vm이 해석할수 있는 IR(Intermediate Representation, 중간언어)를 사용해 1차 컴파일되며 이를 특정 cpu를 target으로 하는 interpreter가 해석해 binary로 만들어낸다
      - IR이 있는 이유?
        1. 문법 검사의 레이어 분리
        2. 컴퓨터가 읽기 쉬운 형태로 재조립
        3. 서로 다른 system call의 환경을 추상화 하기위해(고급언어의 편의성을 위해 사용된 표현을 cpu가 이해하기 쉬운 단위로 나눈다)
        ```java
        public static int add(int a, int b) { return a+b; };
        ByteCode:
            0: iload_0       // 1단계: 첫 번째 파라미터(a)의 값을 작업 스택에 올려라
            1: iload_1       // 2단계: 두 번째 파라미터(b)의 값을 작업 스택에 올려라
            2: iadd          // 3단계: 스택에 있는 두 정수를 꺼내서 더해라
            3: ireturn       // 4단계: 더한 결과값을 반환해라
        // Intel/AMD CPU (x86-64 아키텍처)
        add eax, ebx   ; eax 레지스터와 ebx 레지스터의 값을 더해라
        // 스마트폰/맥북 ARM 아키텍처
        add r0, r1, r2 ; r1과 r2를 더해서 r0에 저장해라
        ```

## 컴파일러의 작동원리

- 컴파일러는 고수준 언어를 저수준 언어로 번역하는 프로그램이다
  - 컴파일러는 `hello world`프로그램과 근본적으로 다르지 않은 평범한 프로그램이다
  - 단지 복잡도 면에서 볼 때 컴파일러가 훨씬 더 복잡할 뿐, 우리가 다룰수 없는 대단한 프로그램이 아니다
- 동작(C Compiler)
  1. 각각의 토큰 추출
  2. syntax tree 생성(compile error/syntax error 없음을 확정, semantic analysis)
  3. 중간 코드 생성(IR)
  4. 중간 코드를 어셈블리어로 변환
  5. 기계 명령어로 변경(binary)

## 컴퓨터 과학에서 추상화가 중요한 이유

- 컴퓨터 시스템은 기본적으로 추상화위에 구축된다
- cpu는 트랜지스터 여러개로 구성되어있지만 명령어 집합이라는 개념으로 내부 구현을 보호한다
  - 따라서 프로그래머는 트랜지스터의 세부 사항은 전혀 고려할 필요 없이 명령어 집합에 포함된 기계 명령어를 사용하여 CPU에 작업을 지시하면 된다
  - 이 이후의 추상화는 위의 프로그래밍 언어 bottom up의 흐름이다
- 입출력은 file로 추상화 되어있다
  - file을 사용할때 어떻게 저장되는지 세부사항은 전혀 신경쓸 필요가 없다
- 실행중인 program은 process로 추상화 된다
  - 프로그래머는 프로그램을 작성할 때 자신의 프로그램이 CPU를 독점한다고 가정할 수 있기 떄문에 단일 CPU 시스템에서도 수많은 프로세스가 동시에 실행될 수 있다
- 물리 메모리와 파일은 virtual memory로 추상화 된다
  - 물리 메모리의 크기가 서로 다른 겨웅에도 자신의 프로그램이 표준적이고 동일한 크기의 메모리에 독점적으로 접근할 수 있다고 가정할 수 있다
  - 또한 mmap를 사용해 가상 메모리를 사용하더라도 실제 메모리를 읽고 쓰는 것처럼 쉽게 파일을 조작할 수 있다
- 네트워크 프로그래밍은 socket으로 추상화된다
  - 프로그래머는 네트워크 packet이 계층 별로 어떻게 해석되는지, NIC가 어떻게 데이터를 송수신하는지 전혀 신경 쓸 필요가 없다
- 프로세스와 프로세스에 종속적인 실행환경은 container로 추상화 된다
  - 프로그래머는 더 이상 개발 환경과 실제 배포 환경의 차이를 겆어할 필요가 없다
- CPU, OS, application은 vm으로 묶여 추상화된다
  - 프로그래머는 더 이상 예전처럼 여러 가지 하드웨어를 구입하여 직접 운영 체제를 설치하고 프로그램을 구성하고 서버 환경을 운영할 필요가 없다
  - vm은 일종의 데이터로서 빠르게 복제가 가능하기 때문에 프로그래머는 많은 서버를 유지하고 관리할 수 있다
  - cloud computing을 지원하는 기술이다
- 추상화는 프로그래머를 저수준 계층에서 점점 더 멀어지게 만들고, 점점 더 저수준 계층의 세부사항도 신경 쓸 필요가 없도록 한다
  - 하지만 추상화를 넘어서는 작업이 필요하다면 이해할 필요가 있다
  - 저수준 계층에 대한 이해는 고급 프로그래머를 구분 짓는 특징 중 하나이다
